# Project Constitution

> このドキュメントはプロジェクトの原則と要件定義を記録したものです。

## プロジェクト概要

### ビジョン

Figma のデザインデータをコマンドラインから取得・処理する CLI ツール。
Figma MCP の代替として、AI エージェントがサブプロセス経由で実行し、正確で必要十分なデザイン情報を取得できることを目指す。

### ターゲットユーザー

- **開発者**: Figma のデザインデータをプログラマティックに扱いたい人
- **AI エージェント**: サブプロセスとして CLI を実行し、デザイン情報を取得する LLM ベースのエージェント

### 成功指標

- 自分やチームの Figma ワークフローが改善される
- AI エージェントから利用した際に、正確で必要十分な情報が得られる
- 技術的な学びが得られる（Figma API、CLI 開発）

---

## 核となる原則

### 設計原則

1. **AI エージェントへの情報提供が最優先**: 出力はエージェントが正確に解釈できる形式・粒度であること
2. **シンプルさ**: 迷ったら簡単な方を選ぶ。不要な抽象化をしない
3. **開発速度**: まず動くものを作る。最適化は後から

### トレードオフの判断基準

| 優先する | より | 理由 |
|---------|------|------|
| シンプルさ | 柔軟性 | 過度な拡張性より読みやすさ・理解しやすさを重視 |
| 開発速度 | パフォーマンス | まず動くものを作り、必要に応じて最適化 |

### やってはいけないこと

- **過度な抽象化**: 不要なラッパーや抽象レイヤーを作らない。3回以上繰り返すまで抽象化しない
- **巨大な依存関係**: 依存ライブラリは最小限に保つ。自作できるレベルのものにライブラリを入れない

## 技術スタック

### 採用技術

| カテゴリ | 技術 | 選定理由 |
|---------|------|---------|
| 言語 | TypeScript | 型安全性、Figma API レスポンスの型定義 |
| ランタイム | Node.js | npm 配布しやすい、ユーザー環境を選ばない |
| CLI フレームワーク | citty (unjs) | TS ファースト、超軽量、オブジェクト定義ベースで型推論が効く |
| ビルド | tsup | ゼロ設定で TS をバンドル |
| パッケージマネージャー | pnpm | 高速・ディスク効率 |
| フォーマッタ・リンター | Biome | 高速な Rust 製、フォーマット・リント一体型 |
| テスト | Vitest | 高速・TS ファースト |

### アーキテクチャ

- **機能ベース**: `features/` の下に機能ごとにまとめる
- CLI コマンド → 機能モジュール → Figma API クライアントの層構成

### 出力形式

- デフォルト: 機械可読な JSON 出力（AI エージェント向け）
- `--pretty` フラグ: 人間が読みやすい形式

## コーディング規約

### 命名規則

- 変数・関数: camelCase
- 型・クラス: PascalCase
- ファイル名: TS の一般的な慣習に従う

### コメント

- 言語: 日本語
- "what" ではなく "why" を書く
- public API には JSDoc/TSDoc を書く

### エラーハンドリング

- **Result 型**を使用（neverthrow 等）
- 例外は予期しないエラーのみ。ビジネスロジックのエラーは Result 型で表現

## 非機能要件

### パフォーマンス

- CLI のレスポンス速度を意識する（起動時間、API 呼び出しの効率化）

### セキュリティ

- Figma API トークンの安全な管理（環境変数 or 設定ファイル、Git にコミットしない）

### AI エージェント連携

- サブプロセスとして実行されることを前提に設計
- デフォルトの JSON 出力は安定した構造を保つ（破壊的変更を避ける）
- エラー時も機械可読なエラーメッセージを返す
- exit code で成功/失敗を明確にする
